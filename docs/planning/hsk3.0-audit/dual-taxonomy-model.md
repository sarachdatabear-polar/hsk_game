# Dual-taxonomy data model (HSK 2.0 record-of-truth + HSK 3.0 additive facet)

_Roadmap R2 steps 3–4. Design + tests only — **no migration ships** until Jordan picks a
rollout option (audit §6). This de-risks option 2 (dual-label) so it's a small, safe change
if chosen._

## The problem the audit proved

HSK 2.0 → 3.0 is a **redistribution**, not a rename (audit: most words change band; only
HSK1 is mostly stable). So the taxonomy cannot be *swapped*. But HSK 3.0 band **can** be
carried alongside as extra, read-only information without disturbing anything.

## The model: `h3` as an additive, read-only catalog facet

**System of record stays HSK 2.0.** `lv` (our level) remains the sole driver of scope
selection, pool building, mastery, stickers, saved sessions, and all `nbhsk.*` persisted
keys. Nothing that produces a stored key ever reads HSK 3.0.

**HSK 3.0 is one optional field on the word record:** `h3` ∈ `{1,2,3,4,5,6,"7-9"}`, present
only for words that appear on the official list (6,347 of 13,921; the rest simply omit it).

```
{ h:"高兴", p:"gāo xìng", e:"happy", t:"ดีใจ", lv:2, f:25, ta:6, tt:6, c:1, n:1, h3:"2" }
                                     ^^^^ record of truth                        ^^^^^ additive facet
```

### Why this needs **no** `migrations.js` schema bump

`src/migrations.js` migrations exist for **stored shapes** (localStorage / `nbhsk.*`).
`h3` lives on the **immutable bundled word catalog** (`data/words.js`, rebuilt from source,
never user-written). Adding a new optional field to catalog records:

- changes no `nbhsk.*` key name, value, or shape → no stored data to migrate;
- is read-only and additive → old installs that never read `h3` behave identically;
- is regenerated by `build_game_data.py`, not persisted by the client.

The **invariant** that makes this safe is proven, not asserted — see below.

## Proven invariant (already green)

`test/hsk3-facet-invariant.test.js` (11 tests, passing) demonstrates on a synthetic fixture
that adding `h3` to every word record leaves **identical output** from every
persisted-key-producing pure function:

- `scopeKey` — concrete key strings unchanged (guards the format saved-session & sticker
  keys are built on);
- `cardSessionKey` — saved-session key unchanged even when the pool is built from
  `h3`-carrying records;
- `stickerDefs` — all sticker ids unchanged (scope/milestone/event);
- `mastery` (`recordAnswer`/`isMastered`) — outcomes keyed on `w.h` identical;
- `buildPool` — identical hanzi set selected (reads `lv`, never `h3`).

This is the load-bearing safety proof: **existing scope keys, mastery, stickers, and saved
sessions remain valid because none of them can see `h3`.**

## Wiring (only if Jordan approves option 2)

Deliberately not done yet. When approved, the change is small and localized:

1. `build_game_data.py`: load `docs/planning/hsk3.0-audit/hsk3-mapping.json`
   (`{hanzi: band}`, already generated — 6,347 entries) and set `h3` on each record whose
   `h` is a key. One dict lookup per word; words absent from the map omit `h3`.
2. Rebuild `data/words.{js,json}` and `data/manifest.json`. No `src/` logic change is
   required for the data to exist.
3. **Surfacing** `h3` in the UI (e.g. an optional "HSK 3.0 band" line on the word-detail
   panel shipped in v84) is a separate, opt-in presentation step with its own i18n + test —
   not part of the data model.

### Regenerating the mapping

`build_h3_mapping.py` re-derives `hsk3-mapping.json` from the verified
`hsk30-source.csv` + `product/by-level`. Re-run whenever the catalog or source refreshes.
The join is exact-match on simplified hanzi (variants split on `|`), lowest band wins.

## What this model deliberately does **not** do

- Does **not** move any word between our levels, or relabel our levels as HSK 3.0.
- Does **not** change scope selection, quests, mastery, or progress.
- Does **not** add the 3,486 missing advanced (7–9) words — that's option 3 (content), a
  separate owner-gated project.
- Does **not** touch character/grammar dimensions (word list only; audit §5).
