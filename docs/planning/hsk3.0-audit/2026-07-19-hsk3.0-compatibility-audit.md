# HSK 3.0 compatibility audit — 2026-07-19

_Roadmap R2 (`docs/planning/2026-07-16-next-roadmap.md`). Parallel engineering lane:
audit **before** any migration. Touches no production data, no saved progress._

## TL;DR

- **A verified, MIT-licensed HSK 3.0 (GF0025-2021) word list is archived** and checksum-matches
  the official standard exactly (11,092 words; per-band 500/772/973/1000/1071/1140/5636).
  See [SOURCE-PROVENANCE.md](SOURCE-PROVENANCE.md).
- **HSK 2.0 → 3.0 is a taxonomy overhaul, not a relabel.** Across our catalog only a small
  minority of words keep the same band; the rest move band or aren't on the official list at
  all. **You cannot rename our current levels to HSK 3.0.** This confirms the roadmap's
  "preserve existing learner progress" and "do not relabel exam-frequency rankings" guidance.
- **The real content cost is the advanced band (7–9): 3,486 words we don't have** (we cover
  only 38% of the 5,605-word advanced band). Bands 1–4 are already well covered (83–94%).
- **Thai coverage is 100%** for every existing catalog word — a full-taxonomy switch adds *no*
  Thai debt for words we already ship; new HSK 3.0 additions would need new Thai + audio.
- **Recommendation:** adopt a **dual-taxonomy data model** (keep HSK 2.0 scope keys as the
  system of record for progress; add HSK 3.0 as an additive facet/label). Do **not** migrate
  the catalog. The rollout shape (dual-catalog vs staged 7–9 vs defer) is **Jordan's decision**
  — see §6.

## Method

`diff_hsk30.py` (this dir, reproducible) compares:
- **Ours:** `product/by-level/HSK{1..6}_words-to-remember_bilingual.csv`, keyed on `hanzi`.
  A word is assigned to the **lowest** level file it appears in (its introduction level;
  recycled words repeat across files).
- **HSK 3.0:** the archived `hsk30-source.csv`, keyed on `Simplified` (variants split on `|`),
  band ∈ {1–6, 7-9}.

Comparison is **word-form vs word-form**. Character-list and grammar-syllabus comparison is
out of scope for this pass (that source isn't in the word CSV) — see §5.

substantially versus the 2.0-era corpus, in both directions.

## 5. Open sub-items (not covered this pass)

- **Character-list gap (3,000 chars) & grammar-syllabus gap.** The archived source is the
  *word* list only. A faithful character/grammar comparison needs the standard's separate
  tables and is a follow-up if Jordan wants full 4-dimensional coverage.
- **Audio.** Game audio is the top-2,000 words by frequency (`build_audio.py`). Any HSK 3.0
  additions promoted into play would need audio regeneration for the words that enter the
  top-2,000 frequency window.
- **Traditional forms.** Upstream flags heuristic traditional-variant filtering; irrelevant to
  this simplified/level comparison but relevant if we ever surface traditional characters.

## 6. Decision for Jordan (roadmap R2 step 5 — audit ends here, migration does not start)

The engineering finding is unambiguous: **keep the current catalog as the system of record;
model HSK 3.0 as an additive label, not a replacement.** What remains is a product call on how
far to go. Three options, in ascending cost:

1. **Defer** — ship nothing now; keep this audit on file until the official 2026 exam calendar
   forces HSK 3.0. *Cost: none.* The roadmap notes the official 2026 calendar still lists
   Levels 1–6 separately, so 2.0 is not gone — deferring is defensible.
2. **Dual-label (recommended engineering baseline)** — add a non-destructive HSK 3.0 band
   facet to existing words (6,347 already map cleanly). No new content, no progress migration;
   learners could optionally see "HSK 3.0 band" alongside our level. *Cost: data-model + build
   only (the §-next task's model + tests de-risk exactly this).*
3. **Staged 7–9 expansion** — additionally author the 3,486 missing advanced words
   (frequency-ranked), Thai + audio, as a *new additive tier* that never disturbs HSK 1–6
   progress. *Cost: large content + translation + audio; owner-gated (native Thai, audio).*

**No option requires migrating or relabeling existing progress.** The next engineering task
builds the dual-taxonomy model + pure migration tests for option 2 so it's ready if chosen.
